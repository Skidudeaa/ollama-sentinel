t+1sWHAT YOU GET

Two tools. One loop.

A file watcher that gets sharper over time. A research agent that tells you which lines of your code break when a library moves.

█ THE SENTINEL

Watches every save.
Remembers every finding.

Edit any file in your project. Within seconds, a local Ollama model reviews it. Findings are upserted to a SQLite database, embedded for semantic recall, and the next review knows what you keep getting wrong. After a week, it knows your team's blind spots better than you do.

local modelsqlite memorysemantic recallversioned reviews

Migrations as
impact analysis.

Ask about a library change. Instead of a generic essay, get a ranked list of every call site in your code that breaks: file, line, severity, suggested fix. AST import graphs, web research, and a LangGraph verify loop, glued together so the answer is grounded in your repo.

t+5sQUICK START

Three commands. Then save a file.

Python 3.10+, a running Ollama instance, one config file. That's the boot sequence.

01 · INSTALL

Clone & install the package.

Editable install. Add [research] only if you want the impact-analysis agent.

$ git clone github.com/Skidudeaa/ollama-sentinel
$ cd ollama-sentinel
$ pip install -e .
02 · OLLAMA

Pull a model. Start the server.

Defaults work with gemma3:4b. Any local model exposing the Ollama API works.

$ brew install ollama
$ ollama pull gemma3:4b
$ ollama serve # :11434
03 · RUN

Init config. Watch.

The first run creates ollama-sentinel.yaml if missing. Edit a file. A review appears.

$ ollama-sentinel init
$ ollama-sentinel run
# save any file to trigger a review
t+30sUNDER THE HOOD

The loop that compounds.

Every save makes the next save's review smarter. Here is what happens between the keypress and the markdown file in .ollama_reviews/.

The dashed feedback arc is the whole point. Each saved finding becomes retrievable context for the next review of any related file. Embedding-ranked, token-budgeted, and capped so the prompt stays useful instead of overflowing. Active project guardrails ride the same injection path — scope-filtered into every matching review — and findings they produce carry their provenance. Best-effort everywhere — extraction failure never blocks the review save.

t+1mSURFACE AREA

Seven moves cover the daily loop.

Each fits on one line; defaults are sensible. A fuller surface sits behind them — a findings lifecycle (findings, resolve, fix, surface, prune), corroboration (confirm, incidents, install-hooks), and the guardrail sub-app — but you can go a long way on these.

·──── 01
run
Watch the working directory and review every save. The default mode.
$ ollama-sentinel run -v
·──── 02
review
Review a single file on demand. -m picks a model role.
$ ollama-sentinel review src/auth.py -m security
·──── 03
report
Show recurring violations ranked by frequency. -f json for tooling.
$ ollama-sentinel report -n 3
·──── 04
triage
Pipe tool output (pytest, mypy, traceback). Get a diagnosis with auto-extracted source.
$ pytest 2>&1 | ollama-sentinel triage
·──── 05
dashboard
Live Rich TUI — recent reviews + recurring violations, polled read-only.
$ ollama-sentinel dashboard -r 0.5
·──── 06
init
Drop a sane ollama-sentinel.yaml in the current directory.
$ ollama-sentinel init
·──── 07
guardrail
Author project rules the reviewer checks every time — or promote ones it proposes from recurring patterns.
$ ollama-sentinel guardrail add no-eval -a "Never eval untrusted input."

t+1wFROM PATTERN TO RULE

The patterns become rules the model enforces.

A recurring pattern is a log entry. A guardrail is a named rule the reviewer checks on every future change. Author one by hand on day one — or let the sentinel propose them from shapes it has flagged and corroborated three or more times. This is the Finding → Incident → Pattern rung, closed.

~/proj · ollama-sentinel guardrail candidates 
             Guardrail candidates (2)
┌───┬──────────┬───┬─────────────────────────────────────────────┐
│ # │ Category │ N │ Drafted assertion (editable)                │
├───┼──────────┼───┼─────────────────────────────────────────────┤
│ 1 │ security │ 4 │ Never pass request data into eval/exec.     │
│ 2 │ bug      │ 3 │ Close database sessions in a finally block. │
└───┴──────────┴───┴─────────────────────────────────────────────┘
  Promote with `guardrail promote <#>` or reject with `guardrail reject <#>`.

$ ollama-sentinel guardrail promote 1
  Promoted candidate 1 → guardrail 5: security-pattern (active, source=promoted).
  → injected into every matching review from now on.

Two guarantees keep it honest, not nagware. Curation gates enforcement — nothing becomes a rule without your promote; a candidate is only a proposal. And a rule can't manufacture its own evidence: a guardrail's own flagged findings reinforce a candidate only when backed by a hard signal (a test failure or a fix commit), never a bare opinion — so a rule can't echo itself back into existence. Clustering runs only when you ask for candidates, never on the review path.

t+1moTHE RESEARCH AGENT

Migrations, but ranked by where your code breaks.

Ask the question, get a punch list.

The research agent runs a multi-step LangGraph workflow that ends in a structured impact analysis — not a generic essay. It scans your repo, resolves imports via AST, matches entities, and ranks every call site by severity.

Optional install. Heavier deps (LangChain, Playwright, LlamaIndex) live behind pip install -e ".[research]".

  1. analyzeplan the research approach
  2. searchweb (SerpAPI / DuckDuckGo)
  3. readPlaywright extracts content
  4. code_searchLlamaIndex over your repo
  5. impact_scanAST graph + entity match
  6. synthesizestructured impact report
  7. verifyloop back if grounding is weak

cfgEVERYTHING IS A KNOB

One YAML. Sensible defaults. Strict validation.

Config is loaded through Pydantic v2. Bad URLs, path traversals, and missing default models all fail loud at startup, not at first review.

watch
Directory, recursion, ignore patterns, debounce. Saves shorter than debounce_ms are coalesced.
ollama.models
One per role. default is required. security, perf, anything else is yours to define.
processing
Chunking, concurrency, and the git_diff_mode flag — review only what changed instead of the whole file.
memory
The violation DB. Toggle off to fall back to no recall. Toggle semantic_recall to use exact-path lookup only.
watch:
  directory: "."
  recursive: true
  debounce_ms: 1500
  ignore_patterns:
    - "**/.git/**"
    - "**/node_modules/**"
    - "**/__pycache__/**"

ollama:
  host: "http://localhost:11434"
  models:
    default:        # required
      name: "gemma3:4b"
      temperature: 0.1
    security:       # pick with: -m security
      name: "gemma3:4b"
      system_prompt: >
        You are a security auditor. Focus on
        injection, auth bypass, OWASP Top 10.

processing:
  max_chars_per_chunk: 12000
  max_concurrent_reviews: 3
  git_diff_mode: false

output:
  directory: ".ollama_reviews"
  history:
    enabled: true
    max_versions: 5

memory:
  enabled: true
  semantic_recall: true
  db_path: ".ollama_reviews/memory.db"
