# Retrospective — CB-3 + Phase A + ADR System Bootstrap

**Date:** 2026-05-02
**Work:** Structural recall wiring (direct to master), CB-3 (f426a55..b9f7968),
Phase A (PR #4, 11 commits), v0.2.0 tag, CI matrix, ADR system bootstrap
**Decisions produced:** ADR-0001 through ADR-0008

## What happened

Single extended session. Started with a vision review of the original
brainstorming document. Identified the Finding/Incident schema gap as the
central architectural move. Shipped three pieces of concrete work
(structural recall, CB-3, Phase A) and wrote the v0.2 Incident spec.
Established the ADR system as the project's documentation architecture.

## What we learned

- Independent reviewers (Opus agent on Phase A) catch real bugs that the
  spec author misses. The merge-in-validator bug (§1) would have shipped
  silently and broken pre-registration for any partial-input config. The
  closed-set role enforcement (§5) caught an interaction between §1's fix
  and the lack of role-name sealing that would have made §1 worse in the
  typo case.
- Specs should specify outcomes and constraints, not test mechanisms. The
  CB-3 spec asked for monkeypatch tests on a closure that couldn't be
  unit-constructed. The implementing agent correctly deviated to
  closure-grep guards following existing codebase precedent.
- The "watching it move" motivation is load-bearing for side projects.
  Phase A was prioritized partly because the hot-path embedding swap
  produces visibly better recall — the kind of change you can feel when
  you save a file. That motivation carried the session.
- Adversarial case enumeration belongs at the top of a spec, before the
  schema. The v0.2 Incident spec leads with seven adversarial cases. If
  the Phase A spec had led with adversarial cases, the merge-in-validator
  bug and the closed-set gap would have been caught at spec time, not at
  implementation time.
- Five projects with the same thesis and zero shipped products is a
  pattern. ollama-sentinel is the furthest along because it has the
  smallest infrastructure footprint (local SQLite, no server). The
  projects that died had servers.

## Precedents established

- **Spec deviation tracking.** PRs that deviate from their spec must name
  the deviations explicitly with numbered §N references and a brief
  rationale in the PR description. Established in Phase A PR #4. See
  ADR-0006.
- **Closure-grep guard pattern.** When a closure inside `build_workflow()`
  can't be unit-constructed without heavy deps, test the extracted helper
  exhaustively and add a source-level grep guard for the wiring. Lives in
  `tests/test_research_agent.py`. See ADR-0004.
- **Leaf-module extraction for testability.** Prompt-assembly helpers go
  in leaf modules (`research_agent/core/prompts.py`) rather than in
  workflow.py, to avoid pulling heavy deps into the test path. See
  ADR-0005.
- **Schema-complete pre-registration.** When a config shape will grow in
  future phases, pre-register all keys in the current phase with defaults.
  Use a merge-in-validator to make pre-registration a schema property, not
  an input property. See ADR-0002, ADR-0007.

## What we'd do differently

- Run the adversarial-case enumeration exercise before writing the schema
  section of any spec. The v0.2 spec does this; the Phase A spec didn't.
- Ask the implementing agent explicitly "are there silent additions beyond
  the flagged deviations?" after execution, not just before.
- Read the actual document before reviewing it. (The Cairn review
  hallucinated section headings and praised prose that didn't exist in the
  file. Caught by the user.)
- Set up CI before the first PR, not after the third.
